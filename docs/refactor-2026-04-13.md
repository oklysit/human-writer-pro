markdown# Human Writer Pro — Implementation Plan Refactor
**Date:** 2026-04-13
**Status:** Supersedes relevant sections of `docs/superpowers/plans/2026-04-13-human-writer-pro-mvp.md`
**Trigger:** Pair-review session + VR validation pilot results + discovery that the existing `human-writer` Claude Code skill already contains the Socratic interview pattern and anti-pattern references the product needs.

## Summary of changes

The original 28-task plan was built around VR as the headline metric and five equally-weighted writing modes. Two things invalidated that framing:

1. **The VR validation pilot (n=54, pre-registered) showed VR is a downstream marker of prompt regime, not a causal lever.** A variant at 33.9% VR fails GPTZero; a variant at 29.1% VR passes. The split is perfect by which prompt produced the variant, not by the VR value. See `eval/reports/vr-validation.md` §9 v3 and §13a for the full analysis.

2. **The existing `human-writer` skill already implements the Socratic interview pattern** (one question at a time, sufficient/partial/insufficient assessment, "user's words are sacred" verbatim stitching discipline) as a WGU-assignment workflow. Generalizing it to a product is smaller work than designing it from scratch. `assignment_workflow.md` Phase 2 is the interview spec. `AI_ANTI_PATTERNS.md` + `BANNED_AI_ISMS.md` together are ~150 terms worth of free quality-gate material. `GOLDEN_DATASET.md` contains 11 mode-specific style rules with paired AI/Human examples that port directly into mode prompts.

The product architecture is now a **two-stage pipeline**:

- **Stage 1 — Socratic interview.** AI asks one question at a time, assesses each response (sufficient / partially sufficient / insufficient), pushes back on vague answers with specifics, tracks coverage against mode-specific rubric items. Goal: gather raw user material with coverage and specificity. Not optimized for VR.
- **Stage 2 — Assembly (band-35 prompt regime).** Takes the gathered raw material and produces draft output with heavy verbatim stitching. Goal: publishable prose with VR as a post-hoc diagnostic, not a gate. Success criterion: passes AI-ism regex gate, GPTZero human+mixed ≥ 51 as aspirational target, user accepts the output.

The product is the productized version of the existing `human-writer` skill, generalized beyond WGU assignments, with a web UI and voice-first input.

### Day 2 — Interview Engine + UI (Tasks 9-17)

#### Task 9 (Interview engine orchestration) — REWRITE

**Original:** generic Q&A loop over Anthropic SDK streaming.

**Revised:** implement the Phase 2 Interrogation pattern from `assignment_workflow.md`:

- Engine asks **one question at a time**, never batched.
- After each user response, the engine assesses:
  - **Sufficient** → confirm briefly, advance to next question.
  - **Partially sufficient** → teach inline with reference to mode-specific guidance from GOLDEN_DATASET, ask a follow-up that targets the gap.
  - **Insufficient** → push back, ask the user to expand with a specific example or concrete detail.
- Engine tracks coverage against mode-specific rubric items (e.g., for cover letter: opener hook, credentials, "why this company," specific professional opinion, close).
- Engine emits a `coverage_score` (0-1) representing "how much of the mode's rubric has been adequately answered." This score drives the assembly button gate in Task 17a.
- Engine emits its current assessment state to the UI (e.g., "Question 3 of ~7 — your last answer was partially sufficient, asking a follow-up").

**Estimated additional time:** ~90 minutes over original Task 9 budget. Mostly porting the logic from the skill file, not inventing new logic.

#### Task 16 (Edit chat component) — REWRITE

**Original:** edit chat proposes rewrites based on user feedback.

**Revised:** edit chat is Socratic, not adversarial. When user highlights text and types "this feels off," the chat does NOT propose rewrites. Instead:

- Engine asks **one** targeted question: *"What were you trying to say that this isn't saying?"* or *"What's the specific word that's wrong?"* or *"Can you give me an example of what you'd rather it sound like?"*
- Takes the user's verbatim response.
- Incorporates the new verbatim into a **localized re-stitch** of just the highlighted paragraph (not a full regeneration).
- The new verbatim becomes raw input for the localized assembly, preserving the user's voice in the edit.

This is the load-bearing UX innovation in the product. It distinguishes Human Writer Pro from "ChatGPT but with a textarea." The rule-of-thumb: the tool never rewrites the user's prose on its own authority; it pulls more verbatim out of the user and re-stitches.

#### Task 19 (Regression fixtures) — REWRITE (substantially)

**Original:** 5 synthetic JSON fixtures with raw_interview text.

**Revised:** the unit of reproducibility is the **(interview + prompt + expected baseline) triple**, not the output. Fixture design surfaced during eval-agent pair review:

**Structure:** `eval/regression-fixtures/cl-assembly/` with one subdir per fixture.

**Per fixture:**
- `interview.md` — frozen copy of the `cl-interview.md` from the original application
- `approved-output.md` — the CL the user actually shipped (for reference tone/style diffing)
- `expected-baseline.json` — baseline scores + tolerance + aspirational targets:

```json
{
  "fixture": "shulman-fleming",
  "baseline": {
    "vr5_mean": 0.299,
    "gptzero_human_plus_mixed": 24,
    "ai_ism_count": 3
  },
  "tolerance": {
    "vr5": 0.05,
    "gptzero_human_plus_mixed": 20,
    "ai_ism_count_max_increase": 2
  },
  "aspirational_targets": {
    "vr5_min": 0.20,
    "gptzero_human_plus_mixed_min": 51
  },
  "notes": "Dense technical register; expected to underperform reflective narrative. Tracks as 'known hard case'."
}
```

**Fixtures to create (all 5 real CLs, including the 2 failures):**

1. **cent-capital** (passed, ~32% VR) — reflective narrative baseline.
2. **devry-university** (passed, ~26% VR) — in the pilot's "dead zone," real-world counter-example.
3. **opencall-ai** (failed, 0.6% VR, pre-VR-phase) — tests whether the current prompt retroactively fixes old interviews.
4. **shulman-fleming** (failed, ~30% VR, dense technical register) — known hard case, tracks as "stays at ~24 or improves."
5. **yo-it-consulting** (passed, ~21% VR, near-threshold) — tests whether the tool holds the near-threshold line.

**Rationale for including the 2 failures:** the failures are more valuable as fixtures than the passes because they're the edge cases. Excluding them would overfit the regression suite to the easy wins. The regression signal for Shulman is "stays at ~24 or improves, not degrades" — not "must reach 51."

**Shared prompts directory:** `eval/regression-fixtures/prompts/`
- `band-35-strategy.md` — the exact 2-sentence stitching instruction used in the pilot, versioned.
- `human-writer-academic.md` — link to (or copy of) the fuller human-writer skill prompt.

Versioning the prompt separately makes prompt changes explicit git diffs — you can see exactly when a regression was introduced.

**Estimated time:** ~45 minutes.

#### Task 20 (Regression runner) — REWRITE

**Original:** single-shot runner over synthetic fixtures, prints markdown table, contains `"Target 5-gram VR ≥ 20%"` in the generation prompt (teaching to the test).

**Revised:** three substantial changes.

**Change 1 — k=3 runs per fixture.** Single-shot numbers are noise. Report mean + std on VR and on judge scores. Run-level variance informs tolerance calibration.

**Change 2 — Remove VR targeting from the generation prompt.** Delete `"Target 5-gram VR ≈ 35%"` from the system prompt used by the runner. Keep stylistic instructions about stitching and verbatim preservation, but do not name the metric the output is going to be scored on. The pilot established the prompt works; the regression suite measures whether it *keeps* working, not whether it can be trained to a number.

**Change 3 — Add an LLM-judge rubric pass.** Sonnet 4.6 as judge (different from generator for cleaner separation), scoring on three axes on a 1-5 scale:

- **Voice match** (compared against `approved-output.md` as golden reference)
- **Content fidelity** (uses only facts from the interview, no hallucination)
- **AI-ism count** (from the regex gate)

Report per-axis scores alongside VR. Budget: ~$3-5 total for k=3 × 5 fixtures × judge calls. This is the real quality signal.

**GPTZero handling:** call GPTZero if under the monthly 10,000-word quota; skip gracefully otherwise and mark results as "GPTZero skipped — quota." Do not block the regression on GPTZero availability.

**Script:** `scripts/eval/run-cl-regression.ts`

**Estimated time:** ~90 minutes (increased from original Task 20 budget).

**Files touched:**
- `scripts/eval/run-cl-regression.ts` — new runner using the (interview + prompt + baseline) structure.
- `scripts/eval/llm-judge.ts` — new judge module using Sonnet 4.6.
